# Простое объяснение проекта и кода

Этот файл нужен не для программиста, а для человека, который хочет понять, как устроен проект и как объяснить его на защите.

## Самая простая идея

AssetControl - это как электронный журнал оборудования предприятия.

В обычной жизни можно представить школьную библиотеку:

- есть книги;
- книги выдают ученикам;
- ученик должен вернуть книгу;
- если книга потерялась, это нужно отметить;
- если книга порвалась, её нужно отправить на ремонт;
- библиотекарь должен видеть всю историю.

В этом проекте вместо книг - оборудование: ноутбуки, мониторы, принтеры, серверы.

## Из каких частей состоит проект

Проект состоит из трёх главных частей.

### 1. Frontend

Frontend - это то, что пользователь видит в браузере.

Например:

- кнопки;
- таблицы;
- формы;
- страницы;
- графики;
- меню слева.

Папка:

```text
frontend/src
```

Главные файлы:

| Файл или папка | Простое объяснение |
|---|---|
| `App.tsx` | Список страниц сайта |
| `pages` | Отдельные страницы системы |
| `components` | Повторяемые части интерфейса |
| `context/AuthContext.tsx` | Хранит информацию о входе пользователя |
| `api/axios.ts` | Настройка запросов к backend |

Простая аналогия: frontend - это витрина магазина. Пользователь видит витрину, но не видит склад и бухгалтерию.

### 2. Backend

Backend - это серверная часть.

Она отвечает за важную работу:

- проверяет логин и пароль;
- проверяет права пользователя;
- получает данные из базы;
- сохраняет изменения;
- создаёт отчёты;
- записывает audit log;
- отдаёт данные frontend-части.

Папка:

```text
backend/src
```

Главные файлы и папки:

| Файл или папка | Простое объяснение |
|---|---|
| `app.ts` | Подключает все API-разделы |
| `index.ts` | Запускает backend |
| `routes` | Адреса API, например `/api/equipment` |
| `controllers` | Логика обработки запросов |
| `middleware/auth.ts` | Проверка пользователя и роли |
| `services/pdfReport.service.ts` | Создание красивых PDF-отчётов |
| `metrics.ts` | Метрики для Grafana |

Простая аналогия: backend - это офис за стойкой. Клиент нажимает кнопку на сайте, а backend решает, можно ли выполнить действие и что записать в базу.

### 3. База данных

База данных - это место, где всё хранится.

Папка со схемой:

```text
backend/prisma/schema.prisma
```

Там описаны главные сущности:

| Сущность | Что означает |
|---|---|
| `User` | Пользователь системы |
| `Equipment` | Оборудование |
| `Employee` | Сотрудник предприятия |
| `Issuance` | Выдача оборудования |
| `RepairTicket` | Ремонтная заявка |
| `InventoryCheck` | Инвентаризация |
| `AuditLog` | Журнал действий |
| `Notification` | Уведомление |

Простая аналогия: база данных - это большой шкаф с папками. В одной папке лежат сотрудники, в другой оборудование, в третьей ремонты.

## Как работает вход в систему

1. Пользователь вводит логин и пароль.
2. Backend проверяет, есть ли такой пользователь.
3. Если пароль правильный, backend выдаёт специальный ключ.
4. Frontend сохраняет этот ключ.
5. При следующих запросах frontend показывает этот ключ backend-части.

Этот ключ называется JWT.

Простая аналогия: JWT - это временный пропуск в здание. Если пропуск есть, охрана понимает, кто вы.

## Как работают роли

В системе разные пользователи имеют разные права.

Например:

- администратор может всё;
- менеджер может выдавать оборудование;
- аудитор может смотреть отчёты;
- обычный сотрудник не может менять глобальные данные.

Простая аналогия: в университете студент, преподаватель и декан могут заходить в одно здание, но права у них разные.

## Как работает выдача оборудования

Когда оборудование выдают сотруднику, система делает несколько шагов:

1. Проверяет, существует ли оборудование.
2. Проверяет, можно ли его выдать.
3. Проверяет, существует ли сотрудник.
4. Создаёт запись о выдаче.
5. Меняет статус оборудования на "используется".
6. Закрепляет оборудование за сотрудником.
7. Записывает действие в audit log.

Важно: это делается как одна операция. Если что-то пошло не так, система не оставит данные в половинчатом состоянии.

Простая аналогия: нельзя одновременно отдать человеку ноутбук, но забыть записать это в журнал. Система делает всё вместе.

## Как работает возврат

Когда сотрудник возвращает оборудование:

1. Система находит активную выдачу.
2. Ставит дату возврата.
3. Меняет статус выдачи на "возвращено".
4. Делает оборудование снова доступным.
5. Убирает текущего владельца.
6. Записывает действие в audit log.

## Как работает ремонт

Если оборудование сломалось:

1. Создаётся ремонтная заявка.
2. Указывается причина и приоритет.
3. Оборудование получает статус "в ремонте".
4. После завершения ремонта система возвращает оборудование в доступное состояние или обратно сотруднику.

Простая аналогия: как ремонт автомобиля. Пока машина в сервисе, её нельзя выдать другому человеку.

## Как работает инвентаризация

Инвентаризация - это проверка: оборудование реально находится там, где указано в системе или нет.

Возможные статусы:

| Статус | Простое объяснение |
|---|---|
| FOUND | Найдено |
| MISSING | Не найдено |
| MOVED | Находится в другом месте |
| DAMAGED | Повреждено |

Простая аналогия: как перекличка в группе. По списку студент должен быть в аудитории. Если его нет, это отмечается.

## Что такое Audit log

Audit log - это журнал важных действий.

Он отвечает на вопросы:

- кто сделал действие;
- когда сделал;
- что именно сделал;
- с какой сущностью работал.

Простая аналогия: это как журнал посещений на проходной. Если что-то произошло, можно посмотреть историю.

Пример:

```text
manager выдал ноутбук сотруднику Иванову
admin создал ремонтную заявку
auditor скачал отчёт
```

## Что такое отчёты

В системе есть CSV и PDF.

### CSV

CSV нужен для Excel.

Например, можно открыть список оборудования и дальше самому сортировать, фильтровать, строить таблицы.

### PDF

PDF нужен для официального вида.

Он выглядит как корпоративный документ:

- название отчёта;
- дата генерации;
- кто сформировал отчёт;
- summary-блоки;
- таблица;
- номера страниц;
- footer.

PDF создаётся в backend в файле:

```text
backend/src/services/pdfReport.service.ts
```

Для русских букв используется шрифт Noto Sans, поэтому кириллица отображается нормально.

## Что такое Dashboard

Dashboard - это главная панель с цифрами.

Там видно:

- всего оборудования;
- сколько доступно;
- сколько используется;
- сколько в ремонте;
- сколько просроченных выдач;
- последние действия.

Простая аналогия: dashboard - это приборная панель автомобиля.

## Что такое Grafana

Grafana показывает состояние системы.

Она нужна, чтобы видеть:

- работает ли база;
- есть ли запросы к API;
- есть ли ошибки;
- сколько оборудования в системе;
- сколько активных выдач;
- сколько ремонтов;
- сколько записей в audit log.

Это не главный функционал системы, а демонстрация того, что проект можно наблюдать и контролировать.

## Что такое Docker

Docker нужен, чтобы быстро запустить весь проект одной командой.

Команда:

```bash
docker compose up --build
```

Она запускает:

- базу данных;
- backend;
- frontend;
- Grafana;
- Prometheus;
- pgAdmin.

Простая аналогия: Docker - это коробка, в которой уже лежит всё нужное для запуска.

## Что проверяют тесты

В проекте есть тесты.

Они проверяют:

- вход пользователя;
- роли;
- запрет действий для сотрудника;
- dashboard;
- PDF-отчёты;
- что PDF действительно начинается как PDF-файл;
- что CSV-отчёты не сломались.

Команда:

```bash
cd backend
npm run test
```

## Как коротко объяснить код комиссии

Можно сказать так:

> Проект разделён на frontend, backend и базу данных. Frontend показывает интерфейс пользователю. Backend проверяет права, выполняет операции и работает с базой. PostgreSQL хранит все данные. Prisma помогает безопасно работать с базой. Важные действия записываются в audit log, а отчёты формируются в CSV и PDF.
